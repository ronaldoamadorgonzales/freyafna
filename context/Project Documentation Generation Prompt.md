# ** Project Documentation Generation Prompt **

You are a **technical writer, software architect, and documentation specialist**. Your task is to generate a **complete, professional project documentation package** based on the project details I will provide.

## **1. Project Overview & Objectives**

Produce a clear and structured overview including:

- Project purpose
    
- Business goals and success criteria
    
- In‑scope and out‑of‑scope items
    
- Stakeholders and roles
    
- Assumptions and constraints
    

## **2. Architecture Overview**

Include:

- High‑level system architecture
    
- Backend and frontend stack
    
- Integrations (APIs, external services, databases)
    
- Data flow diagrams
    
- Infrastructure notes (hosting, CI/CD, environments)
    

## **3. Entity Relationship Diagram (ERD)**

Generate a **Mermaid.js ERD** showing:

- All tables
    
- Relationships and cardinality
    
- Primary and foreign keys
    
- Constraints
    

## **4. Data Model Specification**

For each table, document:

- Table name
    
- Attributes (name, type, constraints)
    
- Primary keys
    
- Foreign keys
    
- Indexing recommendations
    
- Data validation rules
    
- Table purpose and usage notes
    

## **5. Coding Standards & Conventions**

Document:

- Naming conventions
    
- Folder structure
    
- API response format
    
- Error handling strategy
    
- Logging and monitoring approach
    

## **6. Security & Compliance**

Include:

- Authentication & authorization model
    
- Data privacy considerations
    
- Backup & recovery strategy
    
- Access control matrix
    
- Sensitive data handling
    

## **7. Project Phases**

Break down the project into phases such as:

- Discovery & Requirements
    
- Data Modeling
    
- Backend Development
    
- Frontend Development
    
- Integration & API Layer
    
- QA & Testing
    
- Deployment
    
- Post‑launch Monitoring
    

For each phase, include:

- Objectives
    
- Deliverables
    
- Dependencies
    
- Risks
    

## **8. Task Breakdown & Timeline**

Provide:

- Detailed list of tasks per phase
    
- Estimated duration per task
    
- Dependencies and critical path
    
- Suggested timeline (Gantt‑style narrative is acceptable)
    

## **9. Completed Tasks**

List:

- Tasks already completed
    
- Completion dates
    
- Notes or blockers encountered
    
- Impact on next phases
    

## **10. Next Steps & Future Improvements**

Include:

- Remaining tasks
    
- Recommended enhancements for future iterations
    
- Technical debt to address
    
- Opportunities for automation, optimization, or refactoring
    
- Suggested improvements to architecture, workflow, or data model
    

## **11. Acceptance Criteria**

Define:

- What “done” means for each major deliverable
    
- Validation method
    
- Sign‑off owner
    

## **12. Change Management**

Document:

- How new requirements will be handled
    
- Version control workflow
    
- Documentation update process
    

## **13. Testing Strategy**

Include:

- Unit testing
    
- Integration testing
    
- End‑to‑end testing
    
- Load/performance testing
    
- UAT process
    

## **14. Deployment Checklist**

Provide:

- Pre‑deployment checks
    
- Environment configuration
    
- Rollback plan
    
- Post‑deployment verification
    

## **15. Glossary**

Define:

- Key terms
    
- Acronyms
    
- Domain‑specific language
    

## **16. Stakeholder Communication Plan**

Include:

- Meeting cadence
    
- Reporting structure
    
- Communication channels
    

## **Instructions for Output**

- Produce the documentation in **clean, structured Markdown**.
    
- Use headings, subheadings, tables, and diagrams where appropriate.
    
- Ensure clarity, completeness, and professional formatting.
    
- The final output should be **ready for use as an official project documentation file**.
